vim-zsh-tmux-tmuxinator打造终极开发和写作环境

2015-08-09
技术

这篇文章讲的究竟是什么?

这是我这两天开发Rails时候使用的一些工具.经过简简单单的几个步骤,就可以打造出对程序员友好的开发环境.以及日常的写作环境.

很久以来,我们在编写文件的时候都会遇到这样的问题:

  • 由于编辑器的丑陋不够狂拽酷炫吊炸天导致的思维枯竭以及在妹子面前的编辑失常.
  • 上一条是我随便吹的
  • 编辑区域和shell命令不能完整的统一.以前的解决方案很简单,直接另开一个窗口,然后不断的alt+tab,终于找到终端,紧接着不断的cd cd cd 然后run一下命令,好.接着alt+tab好几遍才切换回需要编辑的文件,紧接着脑子一乱,妈蛋,我刚刚编辑文件以后的思路是什么来着?
  • 手贱关掉了终端.然后打开终端不断的cd cd cd vim 然后找到原来的编辑文件.

很显然,我们明明没有必要花时间在切换窗口上面.因为不断的切换,导致我们的思路容易中断.

再举个例子,我现在需要编辑rails项目,那么,在这个过程中我需要开6个终端,这6个终端有用来运行console的,有用来运行rails s的.有用来编辑项目本身的3个(三个比较方便,因为还有一个是gem,一个是todo),还有运行测试的终端.如果使用终端,那我就有两个选择.一是开6个tab,二是使用tmux开6个终端.但前者有一个非常大的缺陷,就是反应速度上较慢,每一个终端无法分屏,以及,问题最大的,初始化极其不方便.

很显然,这篇文章就是用来解决这种编辑时候的低效率问题.

也很显然,尽管我采用的工具都是尽量跟随大神的脚步,大神的配置,但依旧不能解决超级懒而且又想省时间的人.

从这篇文章,你大致可以学到什么?(前提是你需要稍微懂一点点Linux命令)

1.使用vim+zsh+tmux+tmuxinator来打磨自己的日常开发与写作工具.
2.我的使用流程以及几个配置.

vim + zsh + tmux + tmuxinator

  • vim 使用 spf13-vim
  • zsh 使用 oh my zsh
  • tmux 直接apt-get 一下就好.
  • tmuxinator 需要ruby环境.你可以直接安装ruby2.2,然后使用gem install 一下,不过我建议使用rvm安装ruby.

稍微美化一下

  • tmux状态栏(需要打字体补丁)
  • zsh 主题切换

从Web框架的历史来说,为什么Rails是非常出色的Web快速开发框架!

涉及到的点有:

1.为什么要使用Web框架?
2.为什么要使用Rails框架?
3.什么时候该使用什么框架?

1.为什么要使用Web框架?

很简单,追求开发效率.来个场景说明一下:

我登录我的新浪weibo,查看我的私信,浏览别人的webo,然后删除我觉得不好的内容.

那,这个过程的本质是什么呢? 就是取来我的weibo数据和别人的weibo数据,接着对数据进行增删改查.对吧?

其实你仔细想想,这个过程和你所了解的Web开发没有什么关系嘛!

如果世界上每一个人都是程序员,只要个终端链接数据库select一下取出数据不就好了?

聪明的你显然已经看出来了,可以这么做,但并不是每个人都是程序员啊!所以,Web开发的程序员就是将这个过程美化成用户友好的网站,屏蔽了客户端的复杂度,降低了用户的门槛.

真正的交流沟通

2015-03-22
思考

0x00 前言+目录

  1. 0x01 我认为真正的交流沟通
  2. 0x02 沟通的层次以及误区
  3. 0x03 常用的沟通技巧
    E. 0x0E 参考链接+扩展链接

我这里的交流沟通,仅仅局限于语言层次,但是这并不代表交流沟通仅仅属于语言层次,还有很多很多的交流技巧是通过神态,语气,态度,实际行动等等的方式.不过,语言上的交流是最常见的.也是很多人常常犯错误的地方.

能够有效的传达自己的观点opinion,感受feeling,以及事实fact的交流沟通就叫真正的有效交流.

0x01 我认为真正的交流沟通

我认为真正的交流沟通,是能够有效的传达自己的观点opinion,感受feeling,以及事实fact的沟通方式.

但掌握这种技能的人毕竟是少数.

因为,这属于非学校知识.而且是需要长期的磨练的

那么这就引出了第一个需要明确的概念 —- 冰山理论

第一个概念 : 冰山理论

冰山只有1/9浮在水面上.

冰山理论:你看到的仅仅是冰山的一角,冰山有8/9是在水下的.

你需要去追究这些冰山之下的东西.

就好像你问喜欢的人有没有对象,如果说没有,就可以立即走开了.

因为,Ta有没有男朋友/女朋友是不重要的,Ta的态度是最重要的.

当然,多说一句,你去问也不是表白,因为表白是吹响胜利的前兆,是准情人之间的窗户纸的捅破,而如果你被拒绝了,那你这个叫做摊牌,摊牌以后结果必须要由你自己来负责的.

冰山之上 : 观点opinion,感受feeling,以及事实fact

记不清楚自己在哪本书上看到的,把人说的话分解为三种类型.我觉得这个分类实在是太棒了.

它对人的话语进行了如下分类

事实fact - 事实
观点opinion - 观点,基于事实或者感受进行的推断
感受feeling - 感受,情绪

基本上所有的话语都是这三个类型的结合体.

翻一翻自己的聊天记录基本上就可以看出来.

那么,为什么要进行这样子的分类呢?

仅仅是因为,我们要保持一定的理性,不要被自己的不良情绪和别人的混乱逻辑干扰了自己的判断.

区分什么是观点,什么是事实,什么是对方的感觉,这样才可以进行有效的引导或者直接远离脑残(毕竟你没有义务去花时间去让一个脑残明白,他的观点是错误的).

冰山之下 : 推论基础,价值观,道德底线

对这层的理解,我估计是一个顶级的沟通这的修养.

这个就留待我以后讲吧

冰山之外 : 其他补充点.

  1. 不要逞口舌之勇.很多人并没有那么的包容.可能你针对的是论点,但是别人不这么认为.
  2. 讲理是需要双方平等的,在我国这个奇葩的国家,你不要和一个不能容人的领导讲道理.如果有一个非常好的领导,请珍惜.这和谈判一样,双方平等才可以进行谈判.双方不平等谈判那就叫做签订不平等条约.

0x02 沟通的层次以及误区

沟通,我从level上分为四种

  • 基于事实的沟通
  • 基于观点的沟通
  • 基于情绪情感的沟通
  • 基于心灵的沟通

一般情况下,简单关系的人交流的fact ——- 今天天气怎么样?

  • 有才华的人喜欢交流观点,或者技术心得
  • 好朋友和家人交流情绪
  • 亲密爱人和知己交流心灵

从交流的双方对于单个点的态度上,分为三种:

  • 反驳
  • 赞同
  • 瞎扯

反驳的层次

有条有据的反驳是最具有艺术性的活动之一.

<黑客与画家>的作者曾经就写了一个这篇文章.中文名翻译成反驳的七个层次.

我的分类受到这个文章影响最深.
根据黄执中的这篇文章 http://blog.163.com/jonas_hwang/blog/static/2048032302014963956413/

,我们把反驳进行如下分类

  • 谩骂 eg:煞笔
  • 人身攻击 eg:这个人脑子有问题,所以他做的事情一定是错误的.
  • 批评语气和态度 eg:你居然这么说我
  • 无理由反对 eg:我就这么认为.
  • 反驳细节错误 eg:你说的绝大多数是什么意思?有偏差==有犯罪行为?
  • 反驳主要错误 eg:你说的点是这个样子的吧,那好,我来说一组数据
  • 发掘不同见解的背后点,从根源上进行反驳 eg:你可能是担心….我和你一样担心,但是这件事情可能并不是你想象的那个样子.你看…..

对照着这张表,你可以发现,事实上,如果你与其他人的讨论一个社会问题或者其他非专业问题(入门非常低,人人都可以发表自己的看法的问题,你就会发现很多时候,两个人的争吵点完全不在6,7,8三点上.那你就尽量闭嘴)

赞同的层次

正中G点的赞同是最具有艺术性的活动之二.

Feeling:感受 Fact:事实 Compare:对比

补充一句:普通人的一句大神,在大神的眼中并不值得一提.大神只喜欢大神之间的欣赏,一介凡人的欣赏,人家早就麻木了.

瞎扯的层次

无聊的时候闲扯也是最具有艺术性的活动之三.

应该参考这个链接(擦,凸(艹皿艹 ),我果然比较懒,最后自己懒得概述,直接贴链接了).

层次的总结

  • 可以看出来,人与人之间的情感具有可传递性,晓之以理动之以情.不管是夸奖还是反驳.甚至攻击对方也是一样的.
  • 这些技巧需要大量的词汇作为支撑.

0x03 常用的沟通技巧

  • 当需说服一个人的时候,多想想,需要数据支撑吗?需要晓之以理动之以情吗?
  • 当布置一个任务的时候,多想想,我这个任务交代给别人,别人真的会理解我说的每一步吗? 要不然让他再说一遍应该做事情的流程.
  • 不要去直接反驳,而是迂回,想方法找到最实质的原因.而不是在道理上强人一步.
  • 但是你有极少的机会会遇到傻逼,果断逃,这点切记

永远不要和脑残打交道.

0x00 前言(废话)

在平时抓取部分自己喜欢的资源的时候,我们常常会去下载一些比较有用的资源,比如,我比较喜欢GEM的照片,但是,这个东西,总不能总是去找别人要吧,那么,怎么办?

很简单,我们只需要通过Python写一个小小的爬虫就可以解决这些问题.

什么是爬虫?自己可以百度去.

我这里指的爬虫是那些可以模拟浏览器的行为的小程序.

比如,我要抓取G.E.M的相片,那么,我就想个办法.把图片的地址解析出来.然后写一个小功能下载不就好了么.

虽然,话是这么说,

但,怎么下手?
